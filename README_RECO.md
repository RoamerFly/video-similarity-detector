# 视频相似度识别逻辑说明

本文说明当前项目的识别逻辑：程序如何抽帧、如何比较、如何计算 `a_in_b` / `b_in_a`，以及为什么最终会判定为“片段包含”“近似重复”或“不同”。

## 总览

当前识别不是逐像素、逐帧暴力比较，而是下面这条链路：

```text
扫描视频
  -> 预处理画面
  -> 动态抽帧
  -> 为每个视频缓存帧特征
  -> 使用缓存特征做全局候选粗筛
  -> 为每个视频建立 FAISS 帧索引
  -> 对候选视频对做 A->B 和 B->A 双向精确查询
  -> 用时间一致性计算双向覆盖率
  -> 聚合匹配片段和时间窗口
  -> 输出 JSON / CSV / HTML 报告
```

一个关键点：批量分析时，程序会先为每个视频单独抽帧并缓存特征，然后视频对之间复用这些缓存。不是每比较一对视频就重新抽帧。

## 两种模式

### 视频相似度模式

这是默认模式，会执行动态抽帧、CLIP 特征提取、FAISS 检索和双向包含关系判断。

适合判断：

- 一个视频是否是另一个视频的切片。
- 两个视频是否大量重复。
- 两个视频是否只有部分时间段相似。
- 压缩、裁剪、加噪声、轻微调色后的视频是否仍然相似。

### 对比相同文件模式

该模式不抽帧、不加载 CLIP、不比较画面内容。它只用于查找内容完全相同的文件。

流程是：

1. 扫描视频文件。
2. 先按文件大小分组。
3. 只对大小相同的候选文件计算内容指纹。
4. 文件大小和内容指纹都相同，则输出 `identical_file`。

内容指纹是程序内部的快速文件指纹，不是密码学签名。它适合做本地重复文件检查，不等同于视频画面相似度。

## 预处理

每一帧参与 pHash 和 CLIP 特征计算前，都会经过统一几何预处理。对应代码在 `video_sim/preprocess.py`。

处理顺序是：

1. 可选裁剪四周黑边。
2. 如果裁剪后是竖屏，按设置旋转为横屏。
3. 缩放到统一正方形分辨率。

默认配置：

| 参数 | 默认值 | 作用 |
| --- | --- | --- |
| `crop_black_borders` | `true` | 裁剪电影黑边或封装黑边，减少边框干扰。 |
| `resize_mode` | `center_crop` | 等比缩放后居中裁剪成正方形。 |
| `input_size` | `224` | pHash 和 CLIP 使用的统一输入尺寸。 |
| `portrait_rotation` | `right_90` | 竖屏内容默认向右旋转 90 度。 |
| `border_threshold` | `16` | 判断黑边的像素阈值。 |
| `border_crop_ratio` | `0.02` | 边框至少达到画面比例后才裁剪。 |

`resize_mode` 有两种：

- `center_crop`：更关注画面主体，但可能裁掉边缘字幕、UI 或水印。
- `letterbox`：保留完整画面，空余区域补黑边，更适合录屏、字幕、边缘信息重要的视频。

## 动态抽帧

动态抽帧在 `video_sim/frame_sampler.py` 的 `DynamicFrameSampler` 中实现。

程序优先使用 Decord 解码；如果 Decord 失败，会自动回退到 OpenCV。抽帧不是固定每秒一帧，而是基于画面变化决定是否保留。

保留规则：

1. 第一帧总是保留。
2. 当前帧先走同样的预处理流程，再计算 pHash。
3. 当前帧与上一张保留帧的 pHash 相似度如果低于 `skip_threshold`，说明画面变化较大，保留。
4. 如果长时间没有保留帧，达到 `max_gap_sec` 后强制保留一帧。
5. 如果设置了 `frame_step > 1`，则每隔 N 帧检查一次，速度更快但可能跳过短变化。

相关默认值：

| 参数 | 默认值 | 含义 |
| --- | --- | --- |
| `skip_threshold` | `0.90` | 当前帧与上一保留帧相似度大于等于该值时跳过。 |
| `max_gap_sec` | `5` | 最多间隔多少秒强制保留一帧。 |
| `frame_step` | `1` | 每隔多少帧检查一次，`1` 表示逐帧检查。 |

抽帧结果会记录：

- 原视频路径。
- 原始帧号 `frame_index`。
- 时间戳 `timestamp`。
- pHash 字符串。
- 预处理后的 RGB 帧数据，用于后续 CLIP 特征提取。

## 特征缓存

每个视频会生成独立缓存，当前缓存文件名是：

```text
data/video_cache/<视频名_路径hash>/<预处理与抽帧参数profile>/frame_features.npz
```

缓存内容包括：

- `video_path`
- `frame_indices`
- `timestamps`
- `phashes`
- `embeddings`
- `preprocess_config`
- `metadata`

缓存有效性由 metadata 判断，主要包含：

- 视频绝对路径。
- 文件大小。
- 文件修改时间。
- `skip_threshold`
- `max_gap_sec`
- `frame_step`
- 预处理参数。
- embedding 模型名。

如果这些信息不一致，缓存会视为过期。开启“强制重建缓存”会忽略已有缓存并重新抽帧、重新提特征。

注意：当前 `duration_a` / `duration_b` 主要来自缓存里最大保留帧时间戳，是估算时长，不是容器级精确 duration。因为有 `max_gap_sec` 强制保留机制，它通常接近真实时长，但可能比真实时长略短。

## CLIP 特征

帧特征在 `video_sim/embedder.py` 中生成。

当前模型：

```text
openai/clip-vit-base-patch32
```

处理方式：

1. 对每张保留帧提取 CLIP Vision embedding。
2. 每帧得到一个向量。
3. 向量会做 L2 归一化。
4. 缓存到 `frame_features.npz`。

设备选择：

- `auto`：自动检测 CUDA，可用则用 GPU，否则 CPU。
- `cuda`：强制 GPU。
- `cpu`：强制 CPU。

GPU 只影响速度和可用显存压力，理论上不应改变相似度逻辑。

## FAISS 索引

每个视频的帧向量会建立一个 FAISS `IndexFlatIP` 索引。因为向量已 L2 归一化，内积相当于 cosine similarity。

批量分析时：

1. 每个视频只建立一次帧索引。
2. 默认先用缓存中的帧特征建立全局近邻索引，为每个视频挑选有限数量的候选视频。
3. 只有候选视频对进入完整比较；`candidate_limit=0` 时才会比较全部 `N * (N - 1) / 2` 对。
4. 每个候选视频对只进入一次比较流程。
5. 在这一对内部同时计算 A->B 和 B->A 两个方向。

断点恢复签名当前是 `containment_scoring_version = 3`，并会纳入文件指纹、预处理参数、抽帧参数、匹配参数和运行设备。候选粗筛数量不影响单个视频对结果，因此切换候选数量仍可复用已完成的视频对。每个已完成视频对会增量写入 SQLite 断点库，不会每完成一对就重写整份大型状态文件；新版本还会把匹配签名相同的旧大型 JSON 断点一次性迁移到增量库。

## 双向匹配

双向匹配在 `video_sim/matcher.py` 中实现。

设两个视频为 A 和 B：

- A->B：拿 A 的每个保留帧去 B 的 FAISS 索引中找相似帧。
- B->A：拿 B 的每个保留帧去 A 的 FAISS 索引中找相似帧。

每个源帧最多取 `top_k` 个候选，只有 similarity 大于等于 `match_threshold` 的候选才算匹配。

默认值：

| 参数 | 默认值 | 含义 |
| --- | --- | --- |
| `match_threshold` | `0.65` | 帧相似度达到多少才进入匹配列表。 |
| `top_k` | `10` | 每个源帧最多返回多少个候选目标帧。 |

每个匹配点会记录：

- 源视频路径。
- 目标视频路径。
- 源帧号。
- 目标帧号。
- 源时间戳。
- 目标时间戳。
- 帧相似度。

报告里的“当前帧相似度”指的是单个帧匹配的 CLIP cosine similarity，不等于最终视频整体相似度。

## 覆盖率计算

这是当前识别逻辑里最关键的一步。

报告字段：

- `a_in_b`：A 的保留帧中，有多少能在 B 中形成时间一致的匹配。语义是“A 包含于 B”。
- `b_in_a`：B 的保留帧中，有多少能在 A 中形成时间一致的匹配。语义是“B 包含于 A”。
- `symmetric_similarity`：`(a_in_b + b_in_a) / 2`。

注意：`a_in_b` 不是“A 包含 B”，而是“A in B”。如果 `a_in_b` 很高，通常表示 A 可能是 B 的片段。

### 为什么要做时间一致性

只看“有多少帧找到相似帧”会很容易误判。长视频里可能到处出现相似构图、相似人物、相似场景，CLIP 会把这些语义相近但不是同一时间片段的帧也找出来。

因此当前版本不会把所有零散匹配直接累加，而是按时间偏移聚类：

```text
offset = target_timestamp - source_timestamp
```

同一个真实片段的匹配点，通常会共享一个稳定 offset。例如 B 是 A 从 10:00 开始剪出来的片段，那么 B 的时间戳和 A 的时间戳之间会有稳定差值。

当前覆盖率算法：

1. 按 `offset` 排序所有匹配点。
2. 把 offset 差异不超过 `3` 秒的点归为同一簇。
3. 簇内至少需要匹配点，默认最多要求到 `3` 个；源帧很少时会自动降低要求。
4. 对于总保留帧数大于 3 的源视频，簇的源时间跨度或目标时间跨度至少要达到 `1` 秒。
5. 每个簇按 `唯一源帧数量 * 平均相似度` 打分。
6. 只取分数最高的时间一致簇。
7. 覆盖率 = 该簇覆盖的唯一源帧数 / 源视频保留帧总数。

这可以避免“很多零散相似画面”被累计成 100% 覆盖。

## 关系判断规则

最终关系由 `a_in_b`、`b_in_a`、保留帧数和估算时长共同决定。

当前规则顺序如下：

1. 如果 A 明显比 B 长，且 `b_in_a >= 0.65`，判定：

   ```text
   B_is_likely_clip_of_A
   ```

   含义：B 很可能是 A 的片段，也就是 A 包含 B。

2. 如果 B 明显比 A 长，且 `a_in_b >= 0.65`，判定：

   ```text
   A_is_likely_clip_of_B
   ```

   含义：A 很可能是 B 的片段，也就是 B 包含 A。

3. “明显更长”的判断阈值是 `1.35` 倍，时长或保留帧数量满足任一即可。

4. 如果没有明显长短差异，则看方向差：

   ```text
   b_in_a >= 0.75 且 b_in_a - a_in_b >= 0.18
   -> B_is_likely_clip_of_A
   ```

   ```text
   a_in_b >= 0.75 且 a_in_b - b_in_a >= 0.18
   -> A_is_likely_clip_of_B
   ```

5. 如果两边都很高：

   ```text
   a_in_b >= 0.80 且 b_in_a >= 0.80
   -> near_duplicate_or_same_content
   ```

6. 如果任意方向达到部分重叠：

   ```text
   max(a_in_b, b_in_a) >= 0.50
   -> partial_overlap
   ```

7. 其他情况：

   ```text
   different
   ```

### 长视频和短子视频的预期结果

如果 A 是 20 多分钟长视频，B 是几分钟子视频并加了一点噪声，理想情况是：

- `b_in_a` 高：B 的大部分保留帧能在 A 中找到稳定时间偏移匹配。
- `a_in_b` 低或中等：A 的大量额外内容不应都匹配到 B。
- A 的时长或保留帧数明显大于 B。
- 关系应优先输出 `B_is_likely_clip_of_A`，不是 `near_duplicate_or_same_content`。

如果仍然输出近似重复，优先检查：

1. 是否读取了旧报告。
2. 是否复用了旧断点结果。
3. 是否需要重新分析生成 v3 评分结果。
4. 动态抽帧是否过于稀疏，导致长视频和短视频保留帧数量过于接近。
5. `match_threshold` 是否过低，导致语义相似但非同一片段的帧大量进入匹配。

## 匹配片段

匹配片段由 `video_sim/segmenter.py` 聚合。

聚合方式：

1. 合并 A->B 和 B->A 的匹配点。
2. 按 `target_timestamp - source_timestamp` 聚类。
3. offset 差异不超过 `offset_tolerance` 的点归为同一簇。
4. 簇内匹配点数至少达到 `min_segment_matches`。
5. 片段持续时间至少达到 `min_segment_duration`。
6. 去除 A->B / B->A 造成的重复片段。
7. 按 confidence 从高到低排序。

默认值：

| 参数 | 默认值 | 含义 |
| --- | --- | --- |
| `min_segment_duration` | `5` 秒 | 片段至少持续多久才算有效。 |
| `min_segment_matches` | `3` | 片段至少需要多少个匹配点。 |
| `offset_tolerance` | `3` 秒 | 同一片段允许时间偏移波动多少。 |

片段 confidence 由三部分组成：

- 匹配点数量，占 40%。
- 平均相似度，占 40%。
- offset 稳定性，占 20%。

## 时间窗口

时间窗口统计也在 `video_sim/segmenter.py` 中。

当前窗口分别计算：

- A->B 窗口。
- B->A 窗口。

默认 `window_size = 30` 秒。窗口会按源视频时间分段，统计每个窗口内：

- 匹配点数量。
- 匹配比例。
- 平均相似度。
- 最佳目标时间范围。

注意：窗口匹配比例当前按“近似每秒一帧”的假设估算，即 `matched_count / window_size`，不是严格按该窗口真实保留帧数量计算。因此窗口比例更适合辅助定位，不应单独当作最终关系判断。

## 报告字段

主要 JSON 字段：

| 字段 | 含义 |
| --- | --- |
| `video_a` / `video_b` | 视频文件名。 |
| `video_a_path` / `video_b_path` | 视频完整路径。 |
| `a_in_b` | A 包含于 B 的覆盖率。 |
| `b_in_a` | B 包含于 A 的覆盖率。 |

| `segments` | 聚合后的匹配片段。 |
| `windows_a_to_b` / `windows_b_to_a` | 分方向时间窗口统计。 |

当前 UI 提供这些预设：

| 预设 | candidate | skip | match | window | topK | maxGap | step | size | 适合场景 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| 极速 | 6 | 0.60 | 0.58 | 180 | 1 | 45 | 30 | 128 | 海量视频粗筛，最快但最不细。 |
| 快速 | 10 | 0.70 | 0.60 | 120 | 2 | 30 | 16 | 160 | 大批量初筛。 |

## 常见误差来源

### 误报

可能原因：

- `match_threshold` 太低。
- `top_k` 太大。
- 视频里反复出现相似场景、相同人物、相似构图。
- CLIP 更偏语义相似，不是严格像素相同。
- 动态抽帧过稀，导致代表帧不足。

建议：

- 把预设切到“精确”或“完美”。
- 提高 `match_threshold`。
- 使用对比视图查看算法视角帧，确认是否只是语义相似。

### 漏报

可能原因：

- `skip_threshold` 太低，保留帧太少。
- `max_gap_sec` 太大。
- `frame_step` 太大。
- `input_size` 太低，字幕/UI/小目标细节丢失。
- `center_crop` 裁掉了关键边缘内容。

建议：

- 提高 `skip_threshold`。
- 降低 `max_gap_sec`。
- 把 `frame_step` 设为 `1`。
- 提高 `input_size`。
- 尝试 `letterbox`。
- 修改参数后强制重建缓存。

## 对比视图如何看

对比视图有两种视角：

- 原视频视角：使用报告里的帧号优先截取原始视频帧，尽量显示实际画面。
- 算法视角：对同一帧号截取后再执行裁黑边、竖屏旋转、缩放/裁剪，显示算法真正参与 CLIP 比较的画面。

如果两种视角差异很大，通常说明预处理设置改变了画面结构，例如：

- 黑边裁剪裁掉了边缘内容。
- `center_crop` 切掉了字幕或 UI。
- 竖屏旋转方向不合适。
- 匹配分辨率过低导致细节丢失。

## 当前边界

当前算法更擅长“视觉语义相似”和“片段包含”，不是严格版权级取证算法。它不保证：

- 每一帧都严格像素一致。
- 所有加噪声、变速、插帧、镜像、重剪辑都稳定识别。
- 时间窗口覆盖率完全等于真实保留帧覆盖率。
- `duration_a` / `duration_b` 完全等于媒体容器时长。

如果要做最终删除或归档决策，建议结合：

1. 关系标签。
2. `a_in_b` / `b_in_a`。
3. 匹配片段。
4. 对比视图中的原视频视角和算法视角。
5. 必要时切到“精确”或“完美”预设重新分析。
