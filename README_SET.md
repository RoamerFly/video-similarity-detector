# 识别逻辑与参数说明

本文档说明当前项目的视频相似度识别流程、各设置参数的实际作用，以及当识别不准确时优先调整哪些参数。

## 当前识别逻辑

当前“视频相似度识别”不是逐像素逐帧比对，而是“动态抽帧 + 图像特征 + 双向覆盖率”的流程。

1. 扫描视频目录  
   程序递归扫描视频目录下的常见视频文件。每个视频先读取帧数，用于进度估算。

2. 预处理每一帧  
   每个参与计算的画面会按设置进行统一处理：裁剪黑边 -> 竖屏旋转 -> 缩放/裁剪到统一分辨率。这样不同尺寸、不同黑边的视频能放到同一画面尺度下比较。

3. 动态抽帧  
   程序不会固定每秒抽一帧，也不会永远保留每一帧。它会用 pHash 感知哈希比较当前帧和上一张“已保留帧”：
   - 如果当前帧和上一张保留帧足够相似，就跳过。
   - 如果差异超过阈值，就保留。
   - 如果长时间没有保留帧，即使画面变化不大，也会按“最大间隔”强制保留一帧。

4. 提取 CLIP 图像特征  
   对保留下来的帧计算 CLIP 视觉特征，并保存到缓存目录。后续重复分析同一视频、同一抽帧/预处理参数时，可以复用缓存。

5. 全局候选粗筛  
   程序直接读取每个视频已经缓存的帧特征，建立全局近邻索引，先找出每个视频最可能相似的候选视频。明显无关的视频对不会进入昂贵的完整双向比较。把“精确比较候选数”设为 `0` 可关闭粗筛并比较全部视频对。

6. 建立每个视频的帧索引  
   每个视频的帧特征会建立 FAISS 索引。批量比较时，每个视频只建一次索引，然后复用到候选视频对，避免重复抽帧和重复建索引。

7. 双向比较 A -> B 和 B -> A  
   对每一对视频只比较一次，但内部会同时计算两个方向：
   - `A in B`: A 的保留帧中，有多少能在 B 中找到相似帧。
   - `B in A`: B 的保留帧中，有多少能在 A 中找到相似帧。
   这样可以识别“A 是 B 的片段”或“B 是 A 的片段”。

8. 关系判断  
   程序按“双向覆盖率 + 时间一致性 + 时长/保留帧比例”做固定规则判断。覆盖率不是把所有零散相似帧直接累加，而是优先取同一时间偏移链上最稳定的一组匹配；如果 A 明显比 B 更长，且 B 能稳定匹配到 A，即使 A->B 因动态抽帧看起来也很高，也会优先判断为“B 是 A 的片段”，避免长视频和短切片被误判为近似重复。
   - `B_is_likely_clip_of_A`: B 很可能是 A 的片段。
   - `A_is_likely_clip_of_B`: A 很可能是 B 的片段。
   - `near_duplicate_or_same_content`: 两边高度互相覆盖，接近重复或同内容。
   - `partial_overlap`: 有部分重叠。
   - `different`: 暂未识别出明显重叠。

9. 片段和时间窗口  
   匹配到的相似帧会继续聚合为匹配片段，并按固定时间窗口统计局部相似度。片段聚合主要依赖“源时间和目标时间的偏移是否稳定”。

## 当前准确性的边界

识别不准确通常来自以下原因：

- 抽帧太稀：`扫描步长(frame_step)`太大、`跳帧阈值(skip_threshold)`太低、`最大间隔(max_gap_sec)`太大，都可能漏掉短片段或快速变化。
- 匹配太宽松：`匹配阈值(match_threshold)`太低时，画面风格相近但不是同一片段的帧也可能被算作相似。
- 匹配太严格：`匹配阈值(match_threshold)`太高时，压缩、字幕、水印、裁剪、调色后的视频可能匹配不上。
- 分辨率太低：`匹配分辨率(input_size)`太低会丢细节，尤其是字幕、界面录屏、小目标画面。
- 裁剪方式不合适：`center_crop`可能切掉边缘内容；`letterbox`保留完整画面，但可能引入边框区域。
- CLIP 是视觉语义特征：它更擅长判断“画面内容是否相似”，不是严格的逐像素校验。对同场景、同人物、相似构图的视频，可能比传统像素方法更宽松。
- 时间窗口覆盖率是估算值：窗口统计假设近似按时间分布，保留帧密度会影响局部窗口覆盖率。

如果目标是判断“完全相同文件”，请使用“对比相同文件”预设。该模式不抽帧、不跑 CLIP，而是按文件大小和内容指纹判断是否为同一个文件。

## 基础设置

| 设置 | 作用 | 影响 |
| --- | --- | --- |
| Python 路径 | 运行 Python 分析脚本的解释器。打包版通常使用内置 `env`。 | 环境错误会导致无法分析。 |
| 项目目录 | 程序运行基准目录。打包版通常是 exe 同级目录。 | 只读，由程序运行位置决定。 |
| 视频目录 | 要扫描和分析的视频文件夹。 | 分析输入。 |
| 缓存目录 | 保存抽帧、特征、断点恢复等缓存。 | 影响复用速度；清空后会重新抽帧和提特征。 |
| 报告目录 | 保存 JSON、CSV、HTML 报告。 | 结果页和报告页读取这里。 |
| 打开时最大化窗口 | 启动时自动最大化。 | 只影响界面显示。 |
| 关闭窗口时 | 每次询问、最小化到托盘、或退出程序。 | 只影响窗口行为。 |

## 分析配置参数

### 跳帧阈值(skip_threshold)

动态抽帧时使用。当前帧与上一张保留帧的 pHash 相似度如果大于等于该值，就跳过；低于该值才保留。

- 值越低：越容易跳过，速度更快，但容易漏掉细节。
- 值越高：保留更多帧，准确性更好，但耗时和缓存更大。
- 默认值：`0.82`。
- 不准确时建议：漏检短片段时调高到 `0.96` 或 `0.99`。

### 匹配阈值(match_threshold)

CLIP 特征匹配时，两帧被认为相似的最低分数。

- 值越低：召回更多，容易出现误报。
- 值越高：判断更严格，容易漏掉压缩、裁剪、调色后的视频。
- 默认值：`0.64`。
- 不准确时建议：误报多调高到 `0.72` 到 `0.78`；漏报多调低到 `0.60` 到 `0.65`。

### 时间窗口(window_size)

按多少秒切一段统计局部相似度。

- 值越小：更容易定位具体片段，但结果更碎。
- 值越大：更平滑，适合长视频总览，但定位不够细。
- 默认值：`60` 秒。
- 不准确时建议：短视频或短切片用 `10` 到 `15` 秒；长视频粗筛用 `45` 到 `60` 秒。

### 候选数(top_k)

每一帧去目标视频中最多找多少个候选相似帧。

- 值越小：速度更快，但可能错过正确候选。
- 值越大：召回更好，但匹配数量更多，后续片段聚合更慢，也可能带来噪声。
- 默认值：`5`。
- 不准确时建议：漏检时提高到 `20` 或 `32`；误报多时保持 `5` 到 `10`。

### 精确比较候选数(candidate_limit)

全局粗筛后，每个视频最多保留多少个候选视频进入完整双向比较。

- `0`：关闭粗筛，比较全部 `N * (N - 1) / 2` 个视频对，最慢但没有粗筛漏检风险。
- 值越小：大批量任务越快，但较弱的相似关系可能被筛掉。
- 值越大：召回更稳，精确比较数量也会增加。
- 默认值：普通档为 `20`。
- 122 个视频时，普通档通常会把 `7381` 个全组合压缩到约 `1200-2400` 个候选对，实际数量取决于候选关系是否重叠。

### 最大间隔(max_gap_sec)

即使画面一直很相似，也至少每隔多少秒强制保留一帧。

- 值越大：保留帧更少，速度更快。
- 值越小：长静态片段也会保留更多检查点，片段定位更稳。
- 默认值：`18` 秒。
- 不准确时建议：短片段漏检时调到 `2` 到 `3` 秒。

### 扫描步长(frame_step)

动态抽帧阶段每隔多少帧检查一次。

- `1` 表示逐帧检查。
- 值越大：长视频会明显加速，但短变化更容易被跳过。
- 默认值：`6`。
- 不准确时建议：追求准确用 `1`；长视频粗筛可用 `16` 或 `30`。

### 自动裁剪黑边(crop_black_borders)

在计算 pHash 和 CLIP 特征前，先检测并裁剪上下左右黑边。

- 开启：电影黑边、不同封装比例带来的边框影响会减少。
- 关闭：如果视频内容本身有大量黑色 UI 或边缘区域，自动裁剪可能误裁时可关闭。
- 默认值：开启。

### 缩放模式(resize_mode)

把视频帧统一到同一分辨率的方式。

- `center_crop`: 先等比缩放，再居中裁剪成正方形。更利于主体匹配，但可能切掉边缘字幕或 UI。
- `letterbox`: 等比缩放后补边，保留完整画面。更适合界面录屏、字幕、边缘信息重要的视频，但边框可能影响画面特征。
- 默认值：`center_crop`。

### 匹配分辨率(input_size)

预处理后参与 pHash 和 CLIP 特征计算的正方形尺寸。

- 值越低：速度更快、显存更低，但细节损失更大。
- 值越高：细节更多，适合字幕/UI/小目标，但更慢、更吃显存。
- 默认值：`224`。
- 不准确时建议：普通视频 `224`；细节重要用 `256`；最终核验用 `384`。

### 竖屏旋转(portrait_rotation)

如果裁剪黑边后画面是竖屏，按设置方向旋转成横屏后再比较。

- `right_90`: 向右旋转 90 度。
- `left_90`: 向左旋转 90 度。
- 默认值：`right_90`。
- 不准确时建议：竖屏视频方向不一致时切换该项并强制重建缓存。

### 强制重建缓存(force)

忽略已有抽帧和特征缓存，重新处理视频。

- 开启：会重新抽帧、重新计算 CLIP 特征，耗时更久。
- 关闭：能复用同参数缓存，速度快。
- 需要开启的情况：修改了抽帧、预处理、分辨率、旋转方向后，怀疑旧缓存影响结果。

### 运行设备(device)

选择 CLIP 特征提取使用 CPU、CUDA 或自动。

- `auto`: 自动检测 CUDA，能用则用 GPU。
- `cuda`: 强制使用 NVIDIA CUDA。
- `cpu`: 使用 CPU，更慢但稳定性通常更好。
- 不准确性影响：设备本身通常不改变结果，但 CUDA 显存不足可能导致分析失败。

### 最短片段(min_segment_duration)

匹配片段至少持续多久才会进入报告。

- 值越小：能显示更短的相似片段，但噪声也会增加。
- 值越大：过滤零散误报更强，但短切片可能被漏掉。
- 默认值：`5` 秒。

### 最少匹配点(min_segment_matches)

一个片段至少需要多少个相似帧匹配点。

- 值越小：短片段更容易被保留。
- 值越大：片段判断更稳，但需要更多连续证据。
- 默认值：`3`。

### 偏移容忍(offset_tolerance)

聚合同一片段时，源视频和目标视频的时间偏移允许波动多少秒。

- 值越小：时间顺序约束更严格，误聚合更少。
- 值越大：对剪辑、插帧、变速、噪声更宽容，但相似场景可能更容易混在一起。
- 默认值：`3` 秒。

## 预设说明

| 预设 | 主要配置 | 适合场景 | 风险 |
| --- | --- | --- | --- |
| 极速 | candidate `6`, skip `0.60`, match `0.58`, topK `1`, step `30`, size `128` | 海量视频快速粗筛 | 漏检和误报都可能较多 |
| 快速 | candidate `10`, skip `0.70`, match `0.60`, topK `2`, step `16`, size `160` | 大批量初筛 | 较弱关系可能被粗筛过滤 |
| 普通 | candidate `20`, skip `0.82`, match `0.64`, topK `5`, step `6`, size `224` | 日常推荐 | 对强裁剪、字幕、小目标不一定足够 |
| 精确 | candidate `40`, skip `0.92`, match `0.68`, topK `10`, step `3`, size `256` | 结果核验、减少误报 | 更慢，占用更高 |
| 完美 | candidate `0`，其余为最高精度参数 | 少量关键视频最终检查 | 比较全部视频对，最慢且显存压力大 |
| 对比相同文件 | 不抽帧，不跑 CLIP，按文件大小和内容指纹检查 | 找完全相同文件、快速删除重复文件 | 只能识别同一文件，不能识别画面相似或剪辑片段 |

五档相似度预设支持分别保存自定义参数。修改当前档参数后会自动切换并保存到“自定义”，不会覆盖原预设；只有点击“保存到某预设”才会覆盖该档。错误容忍配置也遵循这一规则。“恢复当前预设默认”只重置目标档。

## 不准确时的调参建议

### 漏掉明显相同的片段

优先调整：

1. 把预设切到“精确”。
2. `跳帧阈值(skip_threshold)`调高到 `0.96` 或 `0.99`。
3. `最大间隔(max_gap_sec)`调低到 `2` 到 `3`。
4. `匹配分辨率(input_size)`提高到 `256` 或 `384`。
5. 修改后开启“强制重建缓存”重新分析。

### 误报太多

优先调整：

1. `匹配阈值(match_threshold)`调高到 `0.72` 或 `0.78`。
2. `候选数(top_k)`不要太大，先用 `10`。
3. 如果边缘内容不重要，使用 `center_crop` 并开启裁剪黑边。
4. 在对比视图里查看相似帧，确认是否只是同场景/同风格导致的语义相似。

### 长视频太慢

优先调整：

1. 先用“快速”或“极速”粗筛。
2. 把“精确比较候选数”设为 `6` 到 `20`，避免全部组合精算。
3. `扫描步长(frame_step)`调到 `16` 或 `30`。
4. `匹配分辨率(input_size)`调到 `160` 或 `128`。
5. 粗筛后只对可疑视频用“精确”重新跑。

### 字幕、UI、边缘内容很重要

优先调整：

1. `匹配分辨率(input_size)`提高到 `256` 或 `384`。
2. `缩放模式(resize_mode)`尝试 `letterbox`。
3. 如果自动裁剪误裁 UI 边缘，关闭“自动裁剪黑边”。
4. 修改后强制重建缓存。

### 竖屏视频方向不对

优先调整：

1. 切换“竖屏旋转”为左 90 度或右 90 度。
2. 开启“强制重建缓存”重新分析。
3. 在对比视图中检查左右视频是否方向一致。

## 缓存与参数生效

抽帧和特征缓存与以下参数强相关：

- `skip_threshold`
- `max_gap_sec`
- `frame_step`
- `crop_black_borders`
- `resize_mode`
- `input_size`
- `portrait_rotation`

修改这些参数后，如果结果看起来仍像旧配置，建议开启“强制重建缓存”或在设置页清理缓存后重新分析。

以下参数主要影响匹配或报告，不一定需要重新抽帧：

- `match_threshold`
- `top_k`
- `window_size`
- `min_segment_duration`
- `min_segment_matches`
- `offset_tolerance`

不过断点恢复会记录完整分析签名。如果参数改变，程序会使用新的断点签名，避免把不同配置的比较结果混用。

## 后续可优化方向

当前实现已经能复用每个视频的抽帧和特征缓存，但准确性还可以继续增强：

- 在 UI 中开放 `min_segment_duration`、`min_segment_matches`、`offset_tolerance`。（已采纳）
- 增加“严格像素/结构相似度”二次核验，用于过滤 CLIP 语义误报。
- 对匹配片段按时间顺序做更强约束，减少同场景反复出现导致的错配。
- 时间窗口覆盖率改为基于真实保留帧数量，而不是按秒估算。
- 增加模型选项，例如更适合视频帧/去重任务的视觉 embedding 模型。
